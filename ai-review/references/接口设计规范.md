# 接口设计规范

## 总体目标

- **协议三要素**：接口名、入参、出参清晰自解释
- **存量/增量原则**：存量收拢沉淀，增量开局即规范
- **最终目标**：接口自解释，新成员快速理解

---

## 接口设计原则

### HTTP 方法

| 方法 | 用途 | 要求 |
|-----|------|------|
| GET | 获取资源 | ❌ **禁止用于数据修改** |
| POST | 创建/更新/删除/复杂查询 | 修改操作必须具备幂等性 |

### URI 设计

| 项 | 规范 | 示例 |
|---|------|------|
| 结构 | `/业务域/子业务/操作/{版本号}` | `/order/order/previewOrder/v2` |
| 风格 | 小驼峰 camelCase | ✅ `/previewOrder` ❌ `/preview-order` |
| 查询参数 | `?` 拼接 | `/getOrderDetail/v1?id=111` |

### 版本控制

- URL 嵌入版本号：`/previewOrder/v2`
- 新版本保持兼容，旧版本迁移后下线

## 请求与响应

### 数据格式

| 项 | 规范 |
|---|------|
| 格式 | JSON，`Content-Type: application/json` |
| 日期时间 | `yyyy-MM-dd HH:mm:ss` |
| 金额 | **分**为单位（整数） |
| 字段命名 | 小驼峰 camelCase，同一业务含义保持一致 |

### 请求体结构

```json
// 通用
{
  "orderCode": "11111111111111111111",
  "cancelReason": "不想要了"
}

// 分页
{
  "page": 1,
  "rows": 10,
  "condition": {}
}
```

### 字段命名规范

| 项 | 规范 | 示例 |
|---|------|------|
| 风格 | camelCase | ✅ `userName` ❌ `user-name` |
| 字符 | ASCII 字母/数字/下划线，≤128字符 | `user_id` |
| 语义 | 名词/名词短语，避免缩写 | ✅ `address` ❌ `addr` |
| 后缀 | 禁止 `VO`/`DTO` | ✅ `labelIds` ❌ `LabelVO` |

### 响应体结构

```json
{
  "code": "0",
  "msg": "请求成功!",
  "data": {},
  "trace": null,
  "timestamp": 1751768114432
}
```

| 字段 | 说明 |
|-----|------|
| code | `"0"` 成功，其他为错误码 |
| msg | 提示信息 |
| data | 业务数据 |
| trace | 链路追踪 ID |
| timestamp | 毫秒时间戳 |

## 安全与兼容性

### 敏感数据处理

| 场景 | 要求 |
|-----|------|
| 请求 | 敏感字段加密传输，禁止明文放 URL 参数 |
| 响应 | 脱敏展示，如 `"cardNumber": "6228******1234"` |

### 敏感字段分类

| 类型 | 示例 | 处理要求 |
|-----|------|---------|
| 身份凭证 | 密码、Token、身份证号 | 传输加密，存储哈希 |
| 个人隐私 | 手机号、邮箱、地址 | 传输加密，返回脱敏 |
| 业务敏感 | 订单金额、用户ID | 传输加密，权限控制 |

### 传输安全

- **强制 HTTPS**
- **加密策略**：前端 RSA/SM2 加密 → 后端 SM3 哈希存储

---

## 响应数据设计

| 原则 | 要求 |
|-----|------|
| 最小化返回 | 只返回当前场景需要的数据，DTO 不得直接返回 |
| 避免多层嵌套 | JSON 层级 ≤ 3 层 |
| 空值处理 | 空数组返回 `[]`，空对象可返回 `null` |

---

## 可维护性要求

- **命名自解释**：`previewOrder`、`cancelOrder`，避免缩写
- **字段语义清晰**：`orderStatus`、`payType`，避免 `flag`、`type1`
- **统一性优先**：用户 ID 统一用 `userId`，不得混用 `uid`、`userID`
- **向后兼容**：通过新增字段、增加版本平滑演进
