# SKILL 执行复盘模式

基于执行过程进行复盘优化，支持迭代验证、多轮修改、最终抽象总结的完整优化闭环。

## 场景识别

执行复盘时，首先识别当前属于哪种场景：

| 场景 | 触发条件 | 处理流程 |
|------|----------|----------|
| **首次复盘** | 对话中未进行过 SKILL 相关优化 | 进入「优化模式识别」 |
| **对话中已优化** | 对话中已优化输出，但未修改 SKILL 文件 | 进入「对话优化汇总」 |
| **迭代验证** | 之前已执行过复盘并修改了 SKILL | 进入「迭代优化闭环」 |

---

## 对话优化汇总

用户在对话中已对输出进行优化，但尚未修改 SKILL 文件时：

### 1. 汇总对话中的优化内容

回顾对话历史，提取：
- 用户提出了什么问题/要求
- 做了哪些调整和优化
- 优化后的效果如何
- 用户是否满意

### 2. 判断是否需要沉淀到 SKILL

| 沉淀条件 | 判断依据 |
|----------|----------|
| ✅ 需要沉淀 | 问题具有通用性，未来可能复现 |
| ✅ 需要沉淀 | 优化方式可标准化为规则 |
| ✅ 需要沉淀 | 能补全现有规则的盲区 |
| ❌ 无需沉淀 | 一次性特殊需求 |
| ❌ 无需沉淀 | 与现有规则冲突但属于特例 |
| ❌ 无需沉淀 | 优化内容过于具体，不适合泛化 |

### 3. 输出汇总报告

```markdown
## 对话优化汇总

### 优化内容回顾
- **原始问题**：[用户提出的问题]
- **优化动作**：[做了哪些调整]
- **优化效果**：[效果如何]

### 沉淀判断
- **结论**：[需要沉淀 / 无需沉淀]
- **理由**：[判断依据]

### 建议新增/修改的规则
[如果需要沉淀，列出建议的规则]
```

### 4. 后续操作

- **需要沉淀**：询问用户是否修改 SKILL 文件，确认后进入修改流程
- **无需沉淀**：说明理由，结束本次复盘

---

## 优化模式识别

首次复盘时，根据用户指令选择模式：

| 用户指令关键词 | 优化模式 | 操作 |
|----------------|----------|------|
| `直接优化`、`马上`、`立即` | 直接修改 | 修改 SKILL 文件 + 汇报修改点 + 等待验证 |
| `简要`、`简单`、`概括`、`优化一下` | 精简复盘 | 问题列表 + 修改建议 + 询问是否执行 |
| 其他 | 完整复盘 | 6 步完整输出 + 询问是否优化 |

## 迭代优化闭环

### 场景A：优化后验证

用户执行复盘后进行验证，再次执行时：

1. **读取上次修改记录**：回顾修改了什么、预期解决什么问题
2. **验证结果确认**：询问用户验证结果
   - ✅ 效果理想 → 进入「最终抽象总结」
   - ❌ 效果不理想 → 进入「场景B：效果不理想」
   - 🔄 有新要求 → 进入「场景C：用户新要求」

### 场景B：效果不理想

验证后发现效果未达预期时：

1. **问题定位**：分析上次修改为何未生效
   - 规则未覆盖到该场景
   - 规则表述有歧义
   - 执行时遗漏了某些步骤
2. **追加修改**：针对问题补充或调整规则
3. **记录修改**：追加到修改历史
4. **等待验证**：提示用户再次验证

### 场景C：用户新要求

用户提出新的优化方向时：

1. **理解新要求**：确认用户想要调整的方向
2. **评估影响**：分析新要求与现有规则的冲突或补充关系
3. **修改 SKILL**：调整或新增规则
4. **记录修改**：追加到修改历史
5. **等待验证**：提示用户验证新修改

## 修改历史记录

每次修改 SKILL 后，在对话中维护修改历史：

```markdown
## SKILL 优化记录

### 第N轮修改
- **修改内容**：[具体修改了什么]
- **预期效果**：[解决什么问题]
- **修改片段**：
  ```markdown
  [修改后的规则片段]
  ```

### 第N-1轮修改
...
```

## 复盘6步法

完整复盘时使用：

### 1. 哪些输出没有达到预期？

### 2. 问题分类

属于哪一类：
- **页面发现**：入口、路径、可见性
- **交付完整性**：内容缺失、步骤遗漏
- **工程约束**：技术限制、规范冲突
- **结构完整性**：组织混乱、层次不清
- **消费场景适配**：不符合用户使用场景

### 3. 根因分析

是以下哪种：
- **规则缺失**：没有相关规则
- **规则不明确**：规则模糊、有歧义
- **没有门禁**：缺少检查点
- **上下文过长**：信息过载导致细节丢失

### 4. 应补充的规则

必须使用以下标准格式：

```markdown
### 规则名

- **触发条件**：
- **必做动作**：
- **自检方式**：
- **不通过后果**：

```示例代码```
```

### 5. 修改后的 SKILL 片段

说明这次修改预期解决什么问题。

### 6. 检测类脚本误判分析

如果涉及验证/检测类脚本，检查是否存在：

| 误判类型 | 示例 | 解决方案 |
|----------|------|----------|
| 注释误检 | 注释中的关键词被当作代码 | 排除注释区域 |
| 关键词宽泛 | 部分匹配导致误报 | 使用精确匹配或正则 |
| 上下文忽略 | 未考虑语境约束 | 增加上下文条件 |

## 最终抽象总结

当用户确认优化效果理想后，执行抽象总结：

### 1. 总结本次优化全流程

- 经历了几轮迭代
- 每轮解决了什么问题
- 最终形成了哪些规则

### 2. 抽象通用规则

从本次优化中提取可复用的经验：
- 哪些问题类型可以前置检查
- 哪些规则可以合并或重构
- 是否需要增加新的门禁点

### 3. SKILL 文件最终确认

- 检查 SKILL 文件是否需要精简或重组
- 确认规则表述是否清晰无歧义
- 建议是否需要更新 SKILL 的 description

### 4. 输出优化报告

```markdown
## SKILL 优化报告

### 优化概览
- 迭代轮次：N轮
- 解决问题：X个
- 新增规则：Y条
- 调整规则：Z条

### 规则变更清单
| 规则名 | 变更类型 | 说明 |
|--------|----------|------|
| ... | 新增/修改/删除 | ... |

### 抽象经验
- [可复用的经验1]
- [可复用的经验2]
```
