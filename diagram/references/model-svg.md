# 领域模型图规范

> 📋 **工作流规范**：领域模型图的专属工作流定义在 [SKILL.md](../SKILL.md) 中，执行前请先阅读。
>
> **专属工作流**：分析SQL → 领域划分设计 → 确定四色类型 → 表格化布局方案 → 生成JSON配置 → 执行脚本生成SVG → 预览
>
> **重要说明**：
> - 预览后可手动修改JSON或让LLM调整，重新执行脚本生成SVG
> - 若用户没有提供SQL，则`生成JSON配置`由LLM推理完成，再执行脚本生成SVG

## 快速决策表

| 决策点 | 规则 | 说明 |
|--------|------|------|
| **列数选择** | ≤6模型→3列, 7~12→4列, 13~20→6列, >20→6列 | 优先选多列，减少行数 |
| **布局优先级** | P0:空间利用率 > P1:父子对齐 > P2:连线最短 | 空间利用率最重要 |
| **连线距离** | 父子≤2格, 强关联≤3格, 弱关联≤5格 | 超过需调整位置 |
| **关联表处理** | 纯关联表→简化为属性, 业务明细表→保留为模型 | 看是否有业务属性 |
| **连线数量** | 单模型≤3条 | 超过保留最核心3条 |

## 领域划分设计

1. 基于梳理的表，分析如何合理划分，不宜划分过细，如SCM领域模型图可划分为RBAC域、ERP域、WMS域
2. 若SQL中有不同的表前缀，同一个前缀的通常可考虑划分到同一个域下
3. 使用`ask_user_question`工具与用户确认划分策略

### 确定四色类型

- 从 SQL/文档提取：模型名、英文名、业务属性（过滤审计属性）、核心行为
- 四色分类：角色/资源/描述/时标
- 统计外键关联数，确定中心模型（连线最多原则）
- 检查关联表简化（纯关联表用 `Xxx`/`Xxx集合` 属性表达）

| 类型 | 填充色 | 边框色 | 用途 |
|------|--------|--------|------|
| 时标 | #fce4ec | #e91e63 | 订单/批次/日志记录等 |
| 资源 | #f1f8e9 | #689f38 | 商品/SKU/条码等 |
| 角色 | #fff9c4 | #f9a825 | 用户/租户/供应商等 |
| 描述 | #e3f2fd | #1976d2 | 商品类目/用户分组/OSS配置等 |

### 绘制表格化布局方案

**布局设计工作流**（必须按顺序执行）：

```
1. 统计模型总数 N
2. 根据列数决策表确定 cols
3. 计算最优行列分布（空间利用率优先）
4. 标注父子关系和核心关联
5. 填充模型到网格（先满足空间均匀，再调整关联相邻）
6. 输出表格化布局方案
7. 输出连线关系
```

**统一使用表格化布局方案**：

```markdown
**布局方案**（6列3行，17个模型，利用率94%）：

| 行0 | [品牌](0,0) | [商品分类](0,1) | [商品SPU](0,2) | [SKU规格](0,3) | [SKU条码](0,4) | [SKU物料清单](0,5) |
| 行1 | [供应商商品关联](1,0) | [供应商](1,1) | [订单](1,2) | [订单明细](1,3) | [SKU指定仓库](1,4) | [异常工单](1,5) |
| 行2 | [站点](2,0) | [采购单](2,1) | [采购单明细](2,2) | | | |

**连线验证**：
- ✅ 商品SPU(0,2) → SKU规格(0,3)：同行相邻，距离1格
- ✅ 订单(1,2) → 订单明细(1,3)：同行相邻，距离1格
- ✅ 采购单(2,1) → 采购单明细(2,2)：同行相邻，距离1格
- ✅ 供应商(1,1) → 供应商商品关联(1,0)：同行相邻，距离1格
```

**优势**：
1. **清晰直观**：表格形式一目了然，便于检查空间利用率和模型分布
2. **高效快捷**：不需要绘制复杂的 ASCII 连线图，节省时间
3. **易于验证**：可以快速检查父子关系是否相邻、连线距离是否在限制内
4. **适用所有场景**：无论模型数量多少，表格化布局都清晰可读

**连线验证规范**：

输出布局方案后，必须验证核心连线的距离（示例使用ERP域）：

| 关联类型 | 最大距离 | 验证示例 |
|---------|---------|----------|
| 父子关系 | ≤2格 | ✅ 商品SPU(0,2) → SKU规格(0,3) |
| 强关联 | ≤3格 | ✅ 供应商(1,1) → 供应商商品关联(1,0) |
| 弱关联 | ≤5格 | ✅ SKU规格(0,3) → SKU指定仓库(1,4) |

**域内坐标**：`(行,列)` 格式，如(0,0)表示0行0列，(1,2)表示1行2列。同一个领域内禁止出现重复坐标，否则模型会重叠。

### 领域内布局原则

| 原则 | 说明 | 优先级 |
|------|------|--------|
| **空间利用率优先** | **模型分布尽量均匀，减少空白区域。行数越少越好，每列模型数尽量均衡。空间利用率 > 连线最短** | 🔴 P0 |
| 列均衡 | 每列模型数量尽量均衡，避免某些列很多、某些列很少 | 🔴 P0 |
| 父子对齐 | 强相关的父子关系（如采购单+采购明细、订单+订单明细、SPU+SKU）应放在**同一行或同一列**，保持视觉上的从属关系 | 🟡 P1 |
| 关联就近 | 关联紧密的模型放在相邻位置（同行或同列优先），如连线超过3条，一般放九宫格的中心位置 | 🟡 P1 |
| 连线最短 | 在满足空间利用率的前提下，尽量让有关联的模型相邻，缩短连线距离 | 🟢 P2 |
| 中心居中 | 连线最多的模型放中心位置（但不得以牺牲空间利用率为代价） | 🟢 P2 |
| 方形优先 | N≤4→2x2, N≤9→3x3, N≤16→4x4, N≤25→5x5 | 🟢 P2 |
| 边缘放置 | 游离模型（关联少）放边缘 | 🟢 P2 |
| 关联表不绘制 | 纯关联关系表不作为独立模型 | 🟢 P2 |

**⚠️ 重要：空间利用率优先于连线最短**

实践经验表明，**空白区域过多比连线稍长更影响视觉效果**。布局时应遵循：

1. **先确定最优行列分布**：根据模型总数，选择最紧凑的行列组合
2. **均匀填充每列**：让每列的模型数量尽量接近
3. **在均匀分布的前提下**：调整位置让关联模型相邻

**⚠️ 重要：连线最短仍然是必要约束**

虽然空间利用率优先级更高，但**连线最短仍然是必须遵守的原则**，只是不得以牺牲空间利用率为代价。

**连线距离限制**：

| 关联类型 | 最大距离 | 说明 | 示例 |
|---------|---------|------|------|
| 父子关系 | ≤2格 | 订单+明细、采购单+明细、SPU+SKU **必须相邻或间隔≤1个模型** | ✅ 同行相邻、同列相邻、对角相邻 |
| 强关联 | ≤3格 | 有直接连线的模型，**尽量相邻，最多间隔2个模型** | ✅ 同行/同列/对角邻近 |
| 弱关联 | ≤5格 | 间接关联或可通过中间模型推导的 | ⚠️ 可接受跨1-2行/列 |

**❌ 绝对禁止**：
- 父子模型跨越3格以上（如订单在(0,0)，订单明细在(0,5)）
- 强关联模型放在对角线远端（如(0,0)和(2,5)）
- 为追求空间利用率而完全忽略连线（导致所有连线都很长）

**✅ 正确做法**：
- 在均匀分布的基础上，微调位置让关联模型靠近
- 如果关联模型必须远离，确保不超过上述距离限制
- 优先保证父子关系相邻，再考虑其他关联

**列分布计算示例**：

```
17个模型，6列布局：
❌ 错误方案1（空间利用率56%）：
  列0: 3个  列1: 1个  列2: 2个  列3: 2个  列4: 2个  列5: 3个
  → 大量空白，视觉不美观

❌ 错误方案2（空间利用率83%，但连线过长）：
  行0: [品牌] [商品分类] [商品SPU] [SKU规格] [SKU条码] [供应商]
  行1: [供应商商品关联] [SKU指定仓库] [SKU物料清单] [采购单] [采购单明细] [订单]
  行2: [站点] [异常工单] [订单明细]  ← 订单(1,5)和订单明细(2,2)距离3格，过远！
  → 虽然空间利用率高，但父子模型距离太远

✅ 正确方案（空间利用率83%，连线最优）：
  行0: [品牌] [商品分类] [商品SPU] [SKU规格] [SKU条码] [SKU物料清单]
  行1: [供应商商品关联] [供应商] [订单] [订单明细] [SKU指定仓库] [异常工单]
  行2: [站点] [采购单] [采购单明细]
  → 订单(1,2)和订单明细(1,3)相邻✅ 采购单(2,1)和采购单明细(2,2)相邻✅
```

### 连线精简原则

为避免模型图过于复杂，遵循以下连线规范：

| 规则 | 说明 |
|------|------|
| 核心关联优先 | 只绘制业务上最核心的关联关系 |
| 三线上限 | 单个模型的连线数量不超过3条，超过则保留最核心的3条 |
| 隐式关联省略 | 通过中间模型可间接推导的关联，不直接连线 |
| 自明关联省略 | 命名已明确表达关联的（如xxID），可省略连线 |
| 跨领域不连线 | 不同领域的模型之间不绘制连线 |

示例：若【仓库】需要连接【库位】【安全库存】【入库单】【批次库存】共4条线，则只保留最核心的3条，或调整布局将【仓库】放中心位置减少交叉线。

### 内容过滤

| 类型 | 处理 | 示例 |
|------|------|------|
| 业务属性 | ✅ 保留 | ID、产品编码、备注 |
| 审计属性 | ❌ 移除 | created_at创建时间、updated_by最后修改人、deleted软删除标记 |
| 核心行为 | ✅ 保留 | 上下架、批次追溯（文字精简） |
| 通用行为 | ❌ 不显示 | 新增、修改、查询、删除 |

⚠️ 属性/行为禁止合并：
```
❌ 错误：+ 箱规/托盘规格
✅ 正确：+ 箱规（每行一个）
```

**优先对齐策略**（减少折线）：中心模型放中间列；关联模型同列对齐；高度相近同行。

### 关联关系简化

⚠️ **重要区分**：纯关联表 vs 业务明细表

| 类型 | 判定标准 | 处理方式 |
|------|----------|----------|
| **纯关联表** | 仅含 `id` + 外键 + 审计字段，无业务属性 | 简化为关联属性，不绘制独立模型 |
| **业务明细表** | 含价格、数量、金额、状态等业务属性 | **必须保留为独立模型** |

**必须保留独立模型的表**（含业务属性的明细表）：

| 表类型 | 示例 | 典型业务属性 |
|--------|------|--------------|
| 订单明细 | order_item、erp_order_item | 数量、单价、金额、折扣 |
| 采购明细 | purchase_order_item | 采购数量、已收货数量、单价 |
| 入库明细 | inbound_order_item | 预期数量、实收数量、质检结果 |
| 出库明细 | outbound_order_item | 需求数量、已拣数量 |
| 调拨明细 | transfer_order_item | 计划数量、实际数量、差异数量 |
| 售后明细 | aftersale_item | 售后数量、退货状态、质检结果 |

**可简化为关联属性的表**（纯关联表）：

| 关联表 | 简化方式 | 归属模型 | 判定依据 |
|--------|----------|----------|----------|
| sys_user_role | 属性`角色集合` | 用户 | 仅含 user_id + role_id |
| sys_role_permission | 属性`权限集合` | 角色 | 仅含 role_id + permission_id |
| sys_role_menu | 属性`菜单集合` | 角色 | 仅含 role_id + menu_id |
| wms_wave_order | 属性`出库单集合` | 波次 | 仅含 wave_id + outbound_order_id |

**归属判定**（生命周期分析）：

```
示例：用户 vs 用户角色
用户被删除 → 用户角色随之删除
用户角色被删除 → 用户依然存在
结论：用户角色依附于用户 → 属性挂在【用户】模型下
```

**标记规范**：`Xxx` → 单条关联，`Xxx集合` → 多条关联

## 生成JSON配置

使用 `model_json_generator.py` 从SQL和参数生成JSON配置文件：

```bash
python3 scripts/model_json_generator.py xxx.sql \
  --domains "领域名:表名1,表名2,...;领域名:表名1,..." \
  --models "表名:模型名:类型:行,列:行为、行为:关联属性、关联属性;..." \
  --relations "源模型->目标模型,..." \
  --cols 4 \
  --title XXX领域模型 \
  --json XXX领域模型.json
```

**参数说明**：

| 参数 | 格式 | 示例 |
|------|------|------|
| `--domains` | `领域名:表名1,表名2,...`（多领域用`;`分隔） | `RBAC域:sys_user,sys_role;WMS域:wms_warehouse` |
| `--models` | `表名:模型名:类型:行,列:行为:关联属性` | `sys_user:用户:角色:0,0:登录、分配角色:角色集合` |
| `--relations` | `源模型->目标模型`（多个用`,`分隔） | `租户->用户,用户->角色` |
| `--cols` | 领域列数 | `4` |
| `--title` | 大标题（与JSON文件名对应） | `RBAC领域模型` |
| `--json` | 输出JSON文件 | `RBAC领域模型.json` |

**⚠️ 重要：JSON中relations格式**

生成的JSON配置文件中，`relations` 字段必须使用**字符串数组**格式：

```json
✅ 正确格式：
{
  "relations": [
    "租户->用户",
    "用户->角色",
    "角色->权限"
  ]
}

❌ 错误格式（这是其他图类型的格式，不适用于领域模型图）：
{
  "relations": [
    {"from": "租户", "to": "用户"},
    {"from": "用户", "to": "角色"}
  ]
}
```

**原因**：`model_svg_generator.py` 使用 `'->'` 分割字符串解析连线关系。

## 多领域布局规则

当绘制多个领域时，所有领域**垂直堆叠**排列，宽度统一，高度自适应。

**列数智能决策**

根据**最大领域的模型数量**决定列数：

| 最大域模型数 | 推荐列数 | 行数 | 空间利用率 |
|----------------|----------|------|----------|
| ≤6 | 3 | 1~2行 | 67~100% |
| 7~12 | 4 | 2~3行 | 58~100% |
| 13~20 | 6 | 3~4行 | 72~94% |
| >20 | 6 | 4+行 | 避免过高 |

**⚠️ 重要：列数选择策略**

实践经验表明，**相同模型数下，列数越多，空间利用率通常越高**（因为每行可以放更多模型，减少行数）。

**示例判断**：

```
领域分析：
- RBAC域: 6个模型
- ERP域: 17个模型 ← 最大
- WMS域: 15个模型
- TMS域: 3个模型

最大域=17个模型：
- 按表格推荐：6列
- 验证：6列 → 3行（6+6+5分布，利用率94%）✅
- 对比：5列 → 4行（4+4+4+5分布，利用率85%）⚠️ 多1行
- 结论：选择6列，视觉更紧凑
```

**领域间隔**：各领域外框之间固定间隔 20px

**跨领域连线**：自动过滤，不绘制跨领域的模型连线

## 迭代调整

预览SVG后可通过以下方式调整：

### 手动修改JSON

1. 打开JSON配置文件
2. 修改需要调整的内容（坐标、属性、连线等）
3. 重新执行生成命令

### LLM辅助修改

告诉LLM需要调整的内容，例如：

- 「把用户模型移到中心位置」
- 「删除登录日志模型」
- 「添加创建订单行为到订单模型」
- 「调整连线，只保留核心关联」

LLM会修改JSON配置，然后重新生成SVG。

## 示例

### 完整流程示例

```bash
# 1. 生成JSON配置
python3 scripts/model_json_generator.py init.sql \
  --domains "RBAC域:sys_tenant,sys_user,sys_role,sys_permission,sys_menu,sys_login_log" \
  --models "sys_tenant:租户:角色:0,1;sys_user:用户:角色:1,0:登录、分配角色;sys_role:角色:角色:1,1:分配权限、分配菜单:权限集合、菜单集合;sys_permission:权限:描述:2,0;sys_menu:菜单:描述:2,2;sys_login_log:登录日志:时标:2,1" \
  --relations "租户->用户,租户->角色,用户->角色,角色->权限,角色->菜单,用户->登录日志" \
  --title "RBAC领域模型" \
  --json RBAC领域模型.json

# 2. 从JSON生成SVG
python3 scripts/model_svg_generator.py --json RBAC领域模型.json --svg RBAC领域模型图.svg

# 3. 预览后如需调整，修改JSON后重新生成
python3 scripts/model_svg_generator.py --json RBAC领域模型.json --svg RBAC领域模型图.svg
```

## 命名规范

### 领域

直接使用领域名，不加前缀后缀，长度2-6字符。

| ✅ 正确 | ❌ 错误 |
|--------|--------|

## 问题排查

| 问题现象 | 可能原因 | 解决方法 |
|----------|----------|----------|
| 模型重叠 | 行/列坐标重复 | 确保每个模型的 `行,列` 坐标唯一，脚本不会自动检测冲突 |
| 连线穿过模型 | 关系方向错误 | 调整 `--relations` 中的源->目标 |
| 颜色错误 | 类型值错误 | 确认类型为角色/资源/描述/时标 |
| 缺少行为 | 行为参数未传 | 在 `--models` 中添加行为 |
| 缺少关联属性 | 关联属性参数未传 | 在 `--models` 中添加关联属性 |
| 跨领域连线未显示 | 正常行为 | 跨领域连线自动过滤，只保留领域内连线 |
