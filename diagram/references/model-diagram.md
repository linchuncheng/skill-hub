# 模型图规范

> 本文档定义领域模型图的绘制规范，遵循四色建模法。

## 工作流

使用 `todo_write` 创建任务清单：

分析SQL，提取相关表及字段 → 确定四色类型 → 绘制ASCII布局图及连线关系 → 执行脚本生成SVG → 预览领域模型图

### 确定四色类型

- 从 SQL/文档提取：中文名、英文名、业务属性（过滤审计属性）、核心行为
- 四色分类：角色/事物/配置/单据
- 统计外键关联数，确定中心模型（连线最多原则）
- 检查关联表简化（纯关联表用 `Xxx`/`Xxx集合` 属性表达）

| 类型 | 填充色 | 边框色 | 用途 |
|------|--------|--------|------|
| 角色 | #fff9c4 | #f9a825 | 用户/租户/供应商等 |
| 事物 | #f1f8e9 | #689f38 | 商品/SKU/条码等 |
| 配置 | #e3f2fd | #1976d2 | 商品类目/用户分组/OSS配置等 |
| 单据 | #fce4ec | #e91e63 | 订单/批次/日志记录等 |

### 绘制ASCII布局图及连线关系

用 ASCII 图明确模型位置和连线关系：

```
[存储配置]  [租户]─────────┐
             │           │
  ┌──────────│           │
  ▼          ▼           ▼
[用户] ────→[角色]←────→[权限]
  │          │
  ▼          ▼
[登录日志]  [菜单]

连线关系：租户→用户, 租户→角色, 用户→角色, 角色→权限, 角色→菜单, 用户→登录日志
```

**布局原则**

| 原则 | 说明 |
|------|------|
| 方形优先 | N≤4→2x2, N≤9→3x3, N≤16→4x4, N≤25→5x5 |
| 中心居中 | 连线最多的模型放中心位置 |
| 列均衡 | 每列模型数量尽量均衡 |
| 关联就近 | 关联紧密的模型放在相邻位置或同一列，如连线超过3条，一般放九宫格的中心位置 |
| 边缘放置 | 游离模型（关联少）放边缘 |
| 关联表不绘制 | 纯关联关系表不作为独立模型 |

**连线精简原则**

为避免模型图过于复杂，遵循以下连线规范：

| 规则 | 说明 |
|------|------|
| 核心关联优先 | 只绘制业务上最核心的关联关系 |
| 三线上限 | 单个模型的连线数量不超过3条，超过则不绘制该模型的连线 |
| 隐式关联省略 | 通过中间模型可间接推导的关联，不直接连线 |
| 自明关联省略 | 命名已明确表达关联的（如xxID），可省略连线 |

示例：若【仓库】需要连接【库位】【安全库存】【入库单】【批次库存】共4条线，则只保留最核心的3条，或调整布局将【仓库】放中心位置减少交叉线。

**内容过滤**

| 类型 | 处理 | 示例 |
|------|------|------|
| 业务属性 | ✅ 保留 | ID、产品编码、备注 |
| 审计属性 | ❌ 移除 | created_at、updated_by、deleted |
| 核心行为 | ✅ 保留 | 上下架、批次追溯（文字精简） |
| 通用行为 | ❌ 不显示 | 新增、修改、查询、删除 |

⚠️ 属性/行为禁止合并：
```
❌ 错误：+ 箱规/托盘规格
✅ 正确：+ 箱规（每行一个）
**优先对齐策略**（减少折线）：中心模型放中间列；关联模型同列对齐；高度相近同行。
```

**关联关系简化**

- 纯关联表（无业务属性）处理：仅含 `id` + 外键 + 审计字段时，不绘制为独立模型，用属性形式表达在生命周期更长的模型中。
- 标记规范：`Xxx` → 单条关联，`Xxx集合` → 多条关联
- 归属判定（生命周期分析）：

```
示例：用户 vs 用户角色
用户被删除 → 用户角色随之删除
用户角色被删除 → 用户依然存在
结论：用户角色依附于用户 → 属性挂在【用户】模型下
```

| 关联表 | 简化方式 | 归属模型 | 判定依据 |
|--------|----------|----------|----------|
| sys_user_role | 属性`用户角色集合` | 用户 | 用户删除则关联删除 |
| sys_role_permission | 属性`角色权限集合` | 角色 | 角色删除则关联删除 |

反例：`order_item` 含价格、数量、折扣等业务属性 → 必须作为独立【订单项】模型

### 执行脚本生成SVG

```bash
python3 .qoder/skills/diagram/scripts/model_diagram_preprocessor.py <sql文件1> <sql文件2> ... \
  --models "表名:中文名:类型:行,列:行为:关联属性;..." \
  --relations "源模型->目标模型,..." \
  --context "上下文名称" \
  --svg output.svg
```

| 参数 | 格式 | 示例 |
|------|------|------|
| `--models` | `表名:中文名:类型:行,列:行为:关联属性` | `sys_user:用户:角色:1,1:登录,分配角色:T用户角色` |
| `--relations` | `源模型->目标模型`（使用中文名） | `租户->用户,用户->角色` |
| `--context` | 上下文名称（2-6字符） | `RBAC`、`订单管理` |
| `--svg` | 输出SVG文件 | `RBAC领域模型图.svg` |

**类型值**：`角色` | `事物` | `配置` | `单据`

**关联属性标记**：一对一用 `属性名`，一对多用 `属性名集合`

⚠️ `--relations` 必须使用**中文名**，且与布局规划时的连线关系一致。

完整示例：

```bash
python3 .qoder/skills/diagram/scripts/model_diagram_preprocessor.py init.sql \
  --models "sys_tenant:租户:角色:0,1;sys_user:用户:角色:1,0:登录,分配角色:用户角色;sys_role:角色:角色:1,1:分配权限,分配菜单:角色菜单集合;sys_permission:权限:配置:2,0;sys_menu:菜单:配置:2,2;sys_login_log:登录日志:单据:2,1" \
  --relations "租户->用户,租户->角色,用户->角色,角色->权限,角色->菜单,用户->登录日志" \
  --context "RBAC" \
  --svg RBAC领域模型图.svg
```

## 命名规范

### 上下文名称

直接使用领域名，不加前缀后缀，长度2-6字符。

| ✅ 正确 | ❌ 错误 |
|--------|--------|
| `RBAC` | `供应链SCM-RBAC` |
| `订单管理` | `电商商城平台` |
| `SCM` | `WMS仓储系统` |

### 英文副标题

使用 **PascalCase** 领域概念名，禁止使用数据库表名。

| ✅ 正确 | ❌ 错误 |
|--------|--------|
| `Tenant`, `User`, `OrderItem` | `sys_tenant`, `sys_user`, `order_item` |

### 属性中文映射

| 英文属性 | 中文表述 |
|----------|----------|
| id | ID |
| tenant_id | 租户ID |
| product_code | 商品编码 |
| sku_code | SKU编码 |
| status | 状态 |
| quantity | 数量 |
| batch_no | 批次号 |

## 问题排查

| 问题现象 | 可能原因 | 解决方法 |
|----------|----------|----------|
| 模型重叠 | 行/列配置错误 | 检查 `--models` 中的行,列值 |
| 连线穿过模型 | 关系方向错误 | 调整 `--relations` 中的源->目标 |
| 颜色错误 | 类型值错误 | 确认类型为角色/事物/配置/单据 |
| 缺少行为 | 行为参数未传 | 在 `--models` 中添加行为 |
