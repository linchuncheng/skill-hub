# 核心原理图规范

> 📋 **工作流规范**：核心原理图的专属工作流定义在 [SKILL.md](../SKILL.md) 中,执行前请先阅读。
>
> **专属工作流**：核心思考框架 → 布局意图规划 → 生成SVG(.svg文件) → 验证文件 → 修复循环 → 结果展示

> **公共规范**（配色、动画、SVG技术）详见：[svg-common.md](svg-common.md)

## 本质定义

**核心问题**: 用图形语言讲清楚"这个系统/机制是怎么运作的"

> **与架构图的核心区别**: 架构图回答「有什么」,原理图回答「怎么运作」。核心原理图追求**清晰表达机制**,通过容器嵌套、编号流程、详细标注等手段让读者理解系统内部工作原理。

## 核心思考框架（步骤1:生成前必须回答）

布局不是选模板,而是理解机制后自然形成的结构。生成前先回答这 4 个问题:

| 问题 | 目的 | 示例 |
|------|------|------|
| **主体是谁？** | 明确核心组件/系统/流程 | 消息队列、网关、缓存系统 |
| **怎么流转？** | 数据/消息/状态如何流动 | 生产者→Exchange→Queue→消费者 |
| **关键决策？** | 有哪些分支点、判断条件 | 缓存命中？Yes/No |
| **状态变化？** | 过程中发生了什么转变 | 草稿→审核中→已发布 |

回答完这 4 个问题，布局会自然浮现——流转方向决定主轴，决策点决定分支，状态变化决定层次。

## 设计理念（步骤1.5:必须理解）

1. **确立英雄组件**: 每张图有且只有一个视觉重心，其他元素围绕它展开。英雄组件通常是系统中**数据汇聚、机制最复杂、或最需要被理解**的那个
2. **尺寸传递语义**: 组件大小 = 重要程度 + 内部复杂度。越重要的组件越大，越简单的组件越小。统一尺寸是架构图的做法
3. **呼吸空间是设计出来的**: 空白不是画不满，是让眼睛有地方停顿。大间距 = 阶段切换，小间距 = 同属一组
4. **用视觉暗喻代替文字说明**: 能用图形语言表达的机制，不用文字重复。叠影表示副本、单元格表示 offset、三角表示指针——读者「感受」到机制，而不是「读」到机制
5. **连线样式即语义**: 实线是数据流，虚线是控制/元数据，线宽差异是流量差异。不要让所有连线长得一样
6. **注释只说图说不清的**: 图能表达的，不用文字重复。只有隐藏的条件逻辑、不可见的约束才值得加注释
7. **动画是第二层信息**: 静态结构说明「是什么」，动画说明「什么在动」。流动光点表示数据在传输，脉冲表示端点活跃
8. **有机布局源于有据的不对称**: 同类组件轻微错落（不完全对齐）、行间距不统一（体现阶段感）。有机不是随意，是每一个不对称都有理由

## 布局选型（步骤2）

**核心原理图支持多种布局**,根据场景选择:

| 布局类型 | 适用场景 | 特点 |
|---------|---------|------|
| **中心辐射布局** | 单核心系统、数据汇聚场景 | 英雄组件居中,其他组件环绕 |
| **从左到右流程** | 数据处理流程、请求响应链 | 数据流向清晰,适合步骤编号 |
| **从上到下层次** | 进程模型、调用栈 | 层次分明,体现父子关系 |
| **混合布局** | 复杂系统、多模块协作 | 灵活组合,适应复杂场景 |

> ✅ **推荐使用混合布局**,根据实际机制灵活选择,不必拘泥于单一形式

## 布局设计模式（步骤2）

### 模式1:中心辐射布局

**适用场景**:单核心系统、数据汇聚/分发机制

**设计理念**:
以**英雄组件**为核心,其他组件按交互关系分层环绕:

```
                    [外圈:间接依赖]
                  ↗         ↑         ↖
        [外圈] ←              ↓              → [外圈]
             ↖                ↓                ↗
               ←    [内圈:直接交互]    →
                         ↓
              ┌─────────────────────┐
              │   英雄组件(核心)    │
              │   数据汇聚/机制枢纽   │
              └─────────────────────┘
                         ↓
              [下游:输出/结果]
```

**示例**: Wiki-SOP的LLM智能处理核心

### 模式2:从左到右流程布局

**适用场景**: 数据处理流程、请求响应链、Pipeline机制

**设计理念**:
数据从左到右流动,用编号标注步骤,清晰展示处理过程:

```
[数据源] → [处理1] → [处理2] → [处理3] → [输出]
   ①                  ③                 ⑤
```

**示例**: HDFS文件上传流程(9步编号)、Tomcat Servlet处理(3步编号)

### 模式3:从上到下层次布局

**适用场景**: 进程模型、调用栈、管理层级

**设计理念**:
通过垂直层次体现父子关系、调用关系:

```
          [管理员/客户端]
                 ↓
        ┌────────────────┐
        │   Master/主进程  │
        └────────────────
           ↓      ↓      ↓
      [Worker1][Worker2][Worker3]
```

**示例**: Nginx进程模型、JVM内存层次(内核态→用户态)

### 模式4:混合布局(推荐)

**适用场景**: 复杂系统、多模块协作、多层次架构

**设计理念**:
结合多种布局优势,灵活适应复杂场景:
- 用**大容器**划分系统边界
- 内部用**嵌套容器**组织模块
- 用**编号流程**说明关键步骤
- 用**颜色编码**区分角色/状态

**示例**: Kafka集群、AIoT平台架构

## 生成SVG（步骤3）

### SVG结构模板

核心原理图SVG应包含以下结构:

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 1100 800" width="1100" height="800">

    <!-- 箭头定义 -->
    <marker id="arrow" viewBox="0 0 10 10" refX="9" refY="5" markerWidth="6" markerHeight="6" orient="auto">
      <path d="M 0 0 L 10 5 L 0 10 z" fill="#666"/>
    </marker>
    
    <!-- 动画定义 -->
    <style>
      @keyframes fadeIn {
        from { opacity: 0; }
        to { opacity: 1; }
      }
      .component { animation: fadeIn 0.5s ease-out; }
      .connector { animation: fadeIn 0.8s ease-out; }
    </style>
  </defs>
  
  <!-- 背景 -->
  <rect width="1100" height="800" fill="#fafbfc"/>
  
  <!-- 标题 -->
  <text x="550" y="40" text-anchor="middle" font-size="24" font-weight="bold" fill="#1f2937">图标题</text>
  
  <!-- 组件容器 -->
  <g class="component">
    <!-- 组件矩形 -->

  <!-- 连线 -->
```

### 绘制规则

**1. 组件绘制**:
```xml
<!-- 基础组件 -->
<rect x="X" y="Y" width="W" height="H" rx="6" fill="背景色" stroke="边框色" stroke-width="2"/>
<text x="X+W/2" y="Y+H/2-10" text-anchor="middle" font-size="14" font-weight="bold" fill="文字色">标题</text>
<text x="X+W/2" y="Y+H/2+10" text-anchor="middle" font-size="11" fill="#666">副标题</text>
```

**2. 容器嵌套**:
```xml
<!-- 外层容器(浅色背景) -->
<rect x="50" y="50" width="280" height="280" rx="8" fill="#e3f2fd" stroke="#90caf9" stroke-width="2" stroke-dasharray="5,5"/>
<text x="70" y="80" font-size="14" font-weight="bold" fill="#1565c0">容器标题</text>

<!-- 内部组件 -->
<rect x="70" y="100" width="120" height="60" rx="6" fill="#e1bee7" stroke="#7b1fa2" stroke-width="2"/>
```

**3. 连线绘制(90°折线)**:
```xml
<!-- 水平+垂直折线 -->
<path d="M 240 135 L 300 135 L 300 200 L 380 200" 
      stroke="#7b1fa2" stroke-width="2" fill="none" marker-end="url(#arrow)"/>

<!-- 连线标签 -->
<rect x="280" y="120" width="80" height="20" fill="#fff" rx="3"/>
<text x="320" y="134" text-anchor="middle" font-size="11" fill="#666">1. 数据流</text>
```

**4. 编号流程标注**:
```xml
<!-- 在连线上标注序号 -->
<text x="320" y="134" text-anchor="middle" font-size="11" fill="#7b1fa2" font-weight="bold">① 拉取消息</text>
```

**5. 便签注释**:
```xml
<!-- 黄色便签样式 -->
<g transform="translate(50, 650)">
  <rect width="200" height="80" fill="#fff9c4" rx="4" stroke="#fbc02d" stroke-width="1"/>
  <text x="12" y="25" font-size="12" fill="#666">💡 智能跳过机制:</text>
  <text x="12" y="45" font-size="11" fill="#666">• 无新消息 → 跳过阶段1</text>
  <text x="12" y="62" font-size="11" fill="#666">• 无变更 → 跳过阶段2</text>
</g>
```

### 颜色规范

**组件配色**:
| 角色 | 背景色 | 边框色 | 文字色 |
|------|--------|--------|--------|
| 输入/生产者 | #e1bee7 | #7b1fa2 | #4a148c |
| 输出/消费者 | #ffe0b2 | #e65100 | #bf360c |
| 核心处理 | #c8e6c9 | #2e7d32 | #1b5e20 |
| 存储/数据 | #b2ebf2 | #00838f | #006064 |
| 流程阶段 | #fff9c4 | #f9a825 | #f57f17 |
| 通知/消息 | #f8bbd0 | #c2185b | #880e4f |

**连线配色**:
- 与起点组件边框色一致
- 宽度2px，实线

### 布局原则

1. **英雄组件**: 尺寸最大(240x130)，位置居中或视觉重心
2. **尺寸梯度**: 3个档位 - 大(240x130)、中(180x80)、小(120x60)
3. **呼吸空间**: 阶段边界150-170px，阶段内部30-50px
4. **有机布局**: 允许±10px偏差，不追求完全对齐
5. **文字精简**: 标题≤10字，副标题≤15字，标签≤10字

### 连线规范

**核心原则：优先使用阶段/模块间大箭头，避免组件间细连线**

**大箭头设计（推荐）**：
- ✅ 主流程箭头：5px 粗箭头，带编号标签（如"①数据同步"、"②状态流转"）
- ✅ 输入到阶段：4px 箭头，带功能标签（如"请求输入"、"数据源"）
- ✅ 阶段到输出：4px 箭头，带动作标签（如"持久化"、"通知"）
- ✅ 优势：更清晰、不易出错、视觉层次分明、突出核心链路

**细连线（仅在必要时使用）**：
- 宽度2px，与起点组件边框色一致
- 仅用于组件间的辅助连接，不用于主流程表达

**90°折线规则**（适用于所有连线）：
- ✅ 只能使用水平(`H`)和垂直(`V`)线段
- ✅ 转角必须是90°
- ❌ 禁止斜线
- ❌ 禁止穿过组件

**示例**：
```xml
<!-- 推荐：阶段间大箭头 -->
<path d="M 480 135 H 680" stroke="#2e7d32" stroke-width="5" fill="none" marker-end="url(#arrow-green)"/>
<rect x="550" y="118" width="80" height="24" fill="#fff" rx="4"/>
<text x="590" y="135" text-anchor="middle" font-size="12" font-weight="bold" fill="#2e7d32">①数据同步</text>

<!-- 正确：90°折线 -->
<path d="M 100 100 H 200 V 200 H 300"/>

<!-- 错误：斜线 -->
<path d="M 100 100 L 300 200"/>  ❌
```

**箭头规范**：
- 使用 marker-end，标准尺寸：viewBox="0 0 8 8"，markerWidth="5"，markerHeight="5"
- 箭头路径：`M 0 0 L 8 4 L 0 8 L 1 4 Z`（带凹口V形）
- 参考点：refX="7.5" refY="4"

### 完整示例：多阶段数据处理流程

参考布局（使用阶段间大箭头）：
```
[输入源组]    [阶段1] ══①处理流程═══> [阶段2] ══②流转流程═══> [阶段3]
    │                    │                      │
    └───> 数据源A        └───> 输出目标A         └───> 输出目标B
    └───> 数据源B ══转换处理═══>              └───> 存档目标
    └───> 数据源C ══审核过滤═══>
                              ↓
                        [核心处理]
                              ↓
                        [通知/回调]
```

颜色分配（根据角色）：
- 输入/生产者：紫色系
- 核心处理：绿色系（英雄组件）
- 流程阶段：黄色系
- 输出/消费者：橙色系
- 通知/消息：粉色系
- 存储/数据：青色系

编号流程（在阶段组件内用副标题展示）：
- 阶段1副标题："①接收 ②处理 ③输出"
- 阶段2副标题："①提取 ②转换 ③同步"
- 阶段3副标题："①验证 ②合并 ③发布"

便签注释（说明隐藏机制）：
- 左下角："💡 智能跳过机制..."
- 左中部："🔒 人工审核/门控..."

大箭头设计要点：
- ✅ 主流程用5px粗箭头，突出核心链路
- ✅ 输入/输出用4px箭头，区分层次
- ✅ 所有箭头带白色背景标签，字体加粗
- ✅ 标签文字精简（≤10字），说明流转内容或动作
- ✅ 编号标签使用①②③等序号，明确步骤顺序

## 迭代调整（步骤4-5）

预览SVG后如需调整，直接修改SVG代码：

### 常见调整

**1. 调整组件位置**:
```xml
<!-- 修改 rect 的 x, y 值 -->
<rect x="430" y="280" width="240" height="130" .../>
```

**2. 调整组件尺寸**:
```xml
<!-- 修改 width, height 值 -->
<rect x="430" y="280" width="260" height="140" .../>
<!-- 同时调整文字位置 -->
<text x="560" y="345" .../>  <!-- x = 430+260/2, y = 280+140/2-10 -->
```

**3. 添加/删除组件**:
```xml
<!-- 复制现有组件组，修改坐标和文字 -->
<g class="component">
  <rect x="新X" y="新Y" width="120" height="60" rx="6" fill="#e1bee7" stroke="#7b1fa2" stroke-width="2"/>
  <text x="新X+60" y="新Y+25" text-anchor="middle" font-size="14" font-weight="bold" fill="#4a148c">新组件</text>
</g>
```

**4. 调整连线路径**:
```xml
<!-- 修改 path 的 d 属性 -->
<path d="M 起点X 起点Y H 中间X V 中间Y H 终点X" .../>
```

**5. 修改颜色**:
```xml
<!-- 修改 fill, stroke 值 -->
<rect ... fill="#新背景色" stroke="#新边框色"/>
```

### LLM辅助调整

告诉LLM需要调整的内容，例如：
- 「把LLM核心移到中心位置，尺寸放大到 260x140」
- 「添加一个便签注释在左下角」
- 「调整阶段2的颜色为橙色系」
- 「修改连线标签为编号格式」

LLM会直接修改SVG代码并重新生成。

## 实践要点（步骤5-8:生成时必须遵守）

### 英雄组件确定规则

英雄组件是图中**数据汇聚、机制最复杂、或最需要被理解**的组件,必须：
- ✅ 尺寸最大（比第二大组件大 20-30%）
- ✅ 位置居中或视觉重心
- ✅ 其他元素围绕它展开
- ✅ 动画延迟最后（引导视线最后到达）

### 视觉暗喻：用结构模拟物理形态

| 机制 | 视觉原型 | 实现方式 |
|------|----------|----------|
| 日志/队列 | 长条形色块 + 单元格分割 | 横向长条 + 竖线分割 + 序号标注 |
| 副本/备份 | 叠影效果 | 透明度递减（0.3 → 0.15）+ 微小偏移（+3px） |
| 进度/位移 | 指针标记 | 小三角标注在进度位置 |
| 门控/过滤 | 双边框/漏斗 | 内外两层边框，中间留白 |

### 尺寸梯度设计

- 宽度/高度梯度：按吞吐量、消息量、重要度设置 2-3 个档位（如 165px → 148px → 132px）
- 长度差异：日志/队列按数据量设置长度（如 504px → 420px → 294px）
- 关键原则：差异要有据可依，不是随意设置

### 连线三维语义系统

| 维度 | 语义 | 配置示例 |
|------|------|----------|
| 颜色 | 数据流类型 | 蓝色=写入、青色=读取、紫色=控制 |
| 线宽 | 流量/优先级 | 1.8px=高吞吐、1.6px=中等、1.4px=低 |
| 虚实 | 同步/异步 | 实线=同步数据流、虚线=异步/控制流 |

### 动画延迟策略：引导视线路径

推荐顺序：协调层（0.1s）→ 阶段标签（0.12s）→ 数据源（0.2s）→ 核心组件（0.3s）→ 终点（0.55s）→ 流动光点（1.0s+）

### 呼吸空间：密度传递结构

| 区域 | 间距 | 传递含义 |
|------|------|----------|
| 关键区域内部 | 3-10px | 同属一组，紧密关联 |
| 组件与连线 | 0px | 直接连接，零间隙 |
| 阶段标签与组件 | 20-30px | 关联但不粘连 |
| 阶段边界 | 150-170px | 明确分区，视觉停顿 |

### 阶段标签规范

区域标签直接浮在空白区域,不要用矩形框包围。组件用框（实体）,标签不用框（说明）,读者凭直觉区分「这是什么」和「这叫什么」。

### 组件文字精简规则

组件内的文字以精简为主，**绝不允许文字超出组件边界**。

- 节点标题：≤ 10 字（如 "Producer A"、"Partition 0"）
- 节点副标题：≤ 15 字（如 "Leader @ Broker-0"）
- 标签/注释：≤ 10 字（如 "Key Hash 分区路由"）
- 如果信息太多：拆分到副标题、提取为浮动标签、移到 Footer，**绝对不要**让文字撑破组件

## 验证清单（步骤6:绘制完成后逐项检查）

### 硬性规则（必须通过验证器）
- [ ] 连线不穿过任何组件
- [ ] 连线不悬空（大箭头连接到阶段标签区域，细连线连接到组件四边中点）
- [ ] 文字不溢出组件边界
- [ ] 使用预设配色表
- [ ] 阶段间主流程使用5px粗箭头
- [ ] 所有箭头带白色背景标签

### 柔性规则（LLM 自行检查）
- [ ] 英雄组件是否最大且最突出？
- [ ] 尺寸是否有 2-3 个梯度？（非统一尺寸）
- [ ] 阶段边界是否 150-170px？（视觉停顿）
- [ ] 阶段标签是否未装框？（直接浮动）
- [ ] 连线样式是否传递语义？（颜色+线宽+虚实）
- [ ] 注释是否只说图说不清的？（不重复图已表达的内容）

## 反模式对比

| ❌ 错误做法 | ✅ 正确做法 | 原因 |
|-----------|-----------|------|
| 所有组件统一尺寸（180x70） | 英雄组件 200x90,普通 160x70,辅助 140x60 | 尺寸传递语义 |
| 阶段标签装框 `[阶段1: 收集]` | 阶段标签直接浮动 `阶段1: 收集` | 组件用框（实体）,标签不用框（说明） |
| 所有连线相同样式（2px细线） | 阶段间5px粗箭头+输入/输出4px箭头 | 连线样式即语义，粗箭头突出主流程 |
| 组件间大量细连线 | 阶段间大箭头+必要细连线 | 大箭头更清晰、不易出错 |
| 注释重复图已表达的内容 | 注释只说明隐藏的条件逻辑 | 图能表达的,不用文字重复 |
| 阶段间距均匀（都是 50px） | 阶段边界 150-170px,内部 3-30px | 呼吸空间传递结构 |
